Timeline & network fill guide

Alephium BlockViz — how to read the 3D timeline, gray volumes, cyan planes, and concurrent loads.

1. What you are looking at

  • 16 shards (4×4 group lanes) as cubes along world Z = time (newer toward the live tip).
  • Live tip is the newest open edge of the chain; green outlines mark confirmed tips.
  • Left rail: Network / status / cache. Right: Block inspector. Bottom: timeline minimap.

2. Time grid

4. How network fill works

  1. Live first — force the open 64s tip cell when needed.
  2. Disk — try admit for each lookback k in the visible 7-G ring.
  3. Network history — enqueue incomplete 64s holes in that ring (newest-first bias).
Concurrent interval GETs ≤ 4 (HttpIoPool::kDefaultMaxInflightIntervals). Steady pumps enqueue at most 4 per poll; bootstrap can enqueue more into the job queue but still only 4 run at once.

6. Why so many gray boxes?

Each gray volume is one history subsegment that was enqueued (API queue start), not one cube load.

  • Up to 4 at full opacity ≈ true concurrent HTTP intervals.
  • More boxes at lower opacity = fading after admit or dim extras past the full-α cap.
  • The live tip open cell is never drawn as a gray volume.

If stacks look milky, prefer a side camera view (key 2) so volumes separate along Z.

8. F12 screenshots

F12 requests a GPU readback of the final presentable swapchain image after cubes, debug volumes/planes, and ImGui are drawn, still acquired, before present. It does not copy the MSAA multi-sample buffer.

Files land under docs/images/capture_YYYYMMDD_HHMMSS.png. Opaque UI should match the monitor; translucent overlays are intentionally semi-transparent.
