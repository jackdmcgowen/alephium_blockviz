Network fill policy

How interval GETs are ordered so cubes and gray volumes appear near the camera first.

Enqueue order (each pump)

  1. Live first — when camera is on the live tip segment, force the open 64s cell if needed.
  2. Disk — try admit for lookback k in the visible 7-G ring.
  3. History from camera — map camera Z → genesis-aligned 64s cell (cam_sub), then:
    • enqueue holes from the eye (or resume cursor) walking older,
    • when the crawl hits the admit-ring edge, stop — do not spend budget toward the tip,
    • only after older-in-ring is complete, fill newer toward the live open edge,
    • when the camera catches up / ring expands, resume the older frontier (no index jump).
  4. Dep-critical holes — priority patches after the camera walk, if budget remains.
Concurrent interval GETs ≤ 4 (HttpIoPool::kDefaultMaxInflightIntervals). Multiple 64s holes in the same G may be in flight together so the pool is not stuck on one subseg per window.

What you should see

  • Gray translucent volumes appear first near the camera, then crawl into neighbors.
  • Up to 4 full-opacity volumes ≈ true concurrent HTTP intervals.
  • The live tip open cell is never drawn as a gray history volume.
  • History ranges always stay strictly older than the open live cell.
